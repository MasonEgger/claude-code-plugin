## Efficiency Insights

**What went well:**
- Measured the defect empirically (`wc -c` on the template + a worst-case substitution simulation) instead of eyeballing, which both proved the bug and proved the fix fit. The worst-case substitution (full condition + longest test cmd + long branch) is the right test because the cap is hit at substitution time, not in the template.
- The inline-code ruler trick solves the user's "not swallowed by markdown" constraint exactly: tildes have no meaning inside inline code, so an 80-tilde span is bulletproof, while a bare `~~~~…` line is a GFM tilde fence.

**What could improve:**
- This is yet another orchestrator-string patch (the session-20260531 note predicted bloat would force net-zero edits). This one finally trims rather than adds — but it confirms the prediction: the block had grown past the cap. An adversarial harness (already recommended in that session) would have caught the over-cap emission automatically.

## Process Improvements

- **Treat the 4000-char cap as a build-time assertion, not a hope.** The new Step 3 rule makes the assistant count after substitution; a future harness should also assert worst-case < 4000 by simulating the longest condition/test-cmd/branch, exactly as done here.
- **When framing copyable blocks in markdown, prefer inline code for literal rulers.** Bare repeated `~` or `` ` `` lines get interpreted as fences. Inline code renders them literally and keeps them out of the copy region.
